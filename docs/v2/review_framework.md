# Code Review Framework — Automated Review Instructions

> ⚠️ 本文档是给 自动化审查服务执行用的操作手册，不是给开发者看的。
>
> **每次执行代码审查前**，先读本文档，严格按各 Stage 执行。
> **审查完成后**，将结果写入本地 `meta/review_feedback.md`。该文件是生成输出，不提交到 GitHub。

---

## 0. 必需头信息

每次审查报告必须以如下格式开头：

```markdown
# Code Review Feedback

**Reviewer:** Review Service (configured-provider)
**Date:** YYYY-MM-DD
**Version:** vX.Y.Z（从 versions/README.md 读取最新版本号）
**Branch:** <branch-name>
**Scope:** <本次审查的范围，如：新代码逻辑 / V2 Phase-2 / PR #xxx>
```

版本号读取规则：

1. 打开 `versions/README.md`，找到顶部 `当前版本：` 下列出的最新版本（列表第一项）。
2. 如果同时存在多个版本线（如 v1.2.x 和 v2.0.x），以最高主版本号为准。
3. 将版本号填入 `Version:` 字段。

## 1. 审查流程

### Stage 1 — README 声明验证

目的：确保 `README.md` 描述的功能与代码一致，避免文档落后于实现。

步骤：

1. 读取 `README.md`，提取所有功能声明（bullet points、列举的特性）。
2. 逐条与源码对照验证：
   - 检查对应的宏/函数/状态机是否存在。
   - 检查行为是否符合描述（如 "next-hop 不可达时报 NO_ROUTE"）。
3. 输出格式：

```markdown
### README 声明验证

| # | README 声明 | 源码验证 | 状态 |
|---|------------|----------|------|
| 1 | 描述的功能 | 对应代码位置/逻辑 | ✅ 一致 / ❌ 不一致 / ⚠️ 部分一致 |
```

### Stage 2 — 历史反馈追踪

目的：跟踪之前审查中提出的问题是否已修复，避免重复报告。

步骤：

1. 如本地存在 `meta/review_feedback.md`，读取其中的 **汇总** 或 **历史反馈追踪** 章节；不存在时跳过历史反馈追踪。
2. 逐条验证：
   - 标记为 "Resolved" 的：确认代码是否仍保持修复状态。
   - 标记为 "Open / Warning" 的：重新验证是否已修复。
3. 输出格式：

```markdown
### 历史反馈追踪

| # | 之前的问题 | 状态 | 当前证据 |
|---|-----------|------|----------|
| 1 | 问题描述 | ✅ Resolved / ❌ Still Open | 代码位置或说明 |
```

### Stage 3 — 新代码逻辑审查

目的：深入审查新增/变更代码的逻辑正确性。

审查清单（逐条检查）：

- [ ] **状态机完备性**：枚举值是否覆盖所有状态；状态转换是否有遗漏分支。
- [ ] **竞态条件**：全局变量在多处写入时是否有锁保护或原子操作；中断/回调中修改的数据是否被主循环重复入。
- [ ] **缓冲区安全**：`memcpy_s` / `memset_s` / `snprintf_s` 的长度参数是否正确；是否存在栈上大缓冲区。
- [ ] **空指针与边界**：函数入口是否检查指针参数；数组访问是否检查下标范围。
- [ ] **资源泄漏**：连接断开时是否清理对应的内存/定时器/状态。
- [ ] **整数与类型**：`uint8_t` 溢出场景（循环计数、ID 分配）；有符号/无符号比较混用。
- [ ] **魔法数字**：硬编码数值是否应提取为宏。
- [ ] **协议兼容性**：新增字段是否考虑旧版本解析；包长度是否匹配。

对发现的每个问题，输出格式：

```markdown
- **类型**: Blocker / Warning / Note
- **文件**: `path/file.c:line`
- **问题**: 描述
- **建议**: 修复方案
```

### Stage 4 — 特性验证（如有目标文档）

目的：当审查针对某个版本目标（如 `docs/v2/networking-goal.md`）时，验证实现是否满足文档要求。

步骤：

1. 读取目标文档（如 `docs/v2/networking-goal.md`）。
2. 提取验收标准（通常位于 `验收标准` 或 `目标` 章节）。
3. 逐条对照代码实现。
4. 输出格式：

```markdown
### 特性验证：<文档名>

| # | 验收标准 | 实现位置 | 状态 |
|---|---------|----------|------|
| 1 | 标准描述 | 代码位置/逻辑说明 | ✅ 已实现 / ❌ 未实现 / ⚠️ 部分实现 |
```

### Stage 5 — 汇总

输出本次审查的总结：

```markdown
### 汇总

- **Blocker**: N 个（必须修复）
- **Warning**: N 个（建议修复）
- **Note**: N 个（观察项）
- **Info**: N 个（一般说明）
- **README 声明验证**: N/N 通过
- **特性验证**: N/N 通过

**结论**: ✅ 通过 / ⚠️ 有条件通过 / ❌ 拒绝
```

## 2. 反馈分类定义

| 分类 | 含义 | 处理要求 |
|------|------|---------|
| **Blocker** | 逻辑错误、安全漏洞、协议不兼容，会导致功能失败或数据损坏 | 必须修复后才能合并 |
| **Warning** | 潜在问题：边界条件未覆盖、资源泄漏路径、竞态可能 | 建议修复 |
| **Note** | 设计观察：代码可读性、可维护性、命名建议 | 自行决定 |
| **Info** | 一般性说明：已验证的行为、已确认的设计选择 | 无需操作 |
| **Resolved** | 之前报告的问题现已确认修复 | 无需操作 |

## 3. 反馈文件格式

每次审查结果统一写入本地 `meta/review_feedback.md`，格式：

```markdown
# Code Review Feedback

**Reviewer:** Review Service
**Date:** YYYY-MM-DD
**Version:** vX.Y.Z
**Branch:** <branch-name>
**Scope:** <review-scope>

---

## 1. README 声明验证

...

## 2. 历史反馈追踪

...

## 3. 新代码逻辑审查

...

## 4. 特性验证

...

## 5. 汇总

...
```

写入规则：

- 每次审查**覆盖写入**本地 `meta/review_feedback.md`，只保留本次结果。
- `meta/review_feedback.md` 是生成输出，已加入 `.gitignore`，不作为公开仓库文件保存。

---

## 4. 审查范围决策

每次审查前，根据 Scope 决定执行哪些 Stage：

| Scope | 执行 Stage |
|-------|-----------|
| 全量审查（新仓库/大版本） | 1 + 2 + 3 + 5 |
| 特性验证（针对某个 goal/milestone，如 `docs/v2/networking-goal.md`） | 1 + 4 + 5 |
| Bugfix / PR 审查 | 2 + 3 + 5 |
| 增量/小修改 | 3 + 5 |
| README 同步审查 | 1 + 5 |
